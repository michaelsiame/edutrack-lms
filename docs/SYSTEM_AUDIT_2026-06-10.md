# Edutrack LMS — System Audit (2026-06-10)

Scope: business logic, security, UI/UX, responsiveness, and workflow across the Laravel 10 codebase (170 app PHP files, 154 Blade views, 90 migrations). Findings are ordered by severity within each section. File references are relative to the repo root.

---

## 1. CRITICAL — fix immediately

### 1.1 Production secrets committed to git
`.env` is tracked in version control (`.gitignore` only excludes `.env.backup` / `.env.production`). It contains the production DB password, Gmail SMTP password, Google OAuth client secret, and Lenco API/secret/webhook keys. Commit history even includes `"Restore Google OAuth and Lenco credentials from git history"`, so the secrets exist in multiple commits.

**Fix:**
1. Rotate **all** of these credentials now (DB, Gmail app password, Google OAuth secret, Lenco keys).
2. `git rm --cached .env`, add `.env` to `.gitignore`.
3. Purge history (`git filter-repo` or BFG) if the repo is ever shared, or treat the repo as compromised and rotate regardless.
4. Note: `APP_KEY` in the committed `.env` is empty — confirm the server has a real key set, otherwise sessions/encryption are broken or using a predictable key.

### 1.2 No brute-force protection on login
A `login` rate limiter is defined in `app/Providers/RouteServiceProvider.php:48` but **never applied** — `POST /login` (routes/web.php:46) has no `throttle:login` middleware. The `users` table has `failed_login_attempts` / `account_locked_until` columns (cast in `app/Models/User.php:53-54`) but no code reads or writes them. Unlimited password guessing is possible.

**Fix:** add `->middleware('throttle:login')` to the login POST route, and either implement the account-lockout columns or drop them.

### 1.3 Suspended/inactive users can still log in
`users.status` (`active|inactive|suspended`, managed in `app/Http/Controllers/Admin/UserController.php`) is never checked anywhere — not in `LoginController`, not in `GoogleController::callback`, not in middleware. Suspending a user has no effect; existing sessions and Sanctum tokens also survive.

**Fix:** check `status === 'active'` after `Auth::attempt` and in the Google callback; add a middleware (or extend `Authenticate`) that logs out non-active users on every request.

### 1.4 Sensitive files are world-readable in `public/`
Assignment submissions are stored on the `public` disk (`app/Http/Controllers/Student/AssignmentController.php` → `storage/app/public/assignment-submissions/...`), and `public/uploads/` contains `assignments/`, `payments/`, `certificates/`, `users/`. Anyone with (or guessing) a URL can download other students' submissions, payment proofs, and certificates with no authentication — the controller-level checks only protect the *named* download routes, not the raw files.

**Fix:** move private files to the `local` (non-public) disk and serve them exclusively through authorized controller responses (`Storage::download` after ownership checks). Keep only genuinely public assets (course thumbnails, team photos) on the public disk.

### 1.5 IDOR: cross-course resource download
`app/Http/Controllers/Student/LessonResourceController.php::download` verifies the student is enrolled in `{course}` and that `{resource}` belongs to `{lesson}` — but never that `{lesson}` belongs to `{course}`. Any student enrolled in *any* course can download every resource of every other course by swapping the course segment in the URL.

**Fix:** add the same check used in `LearningController::show`:
```php
if (!$lesson->module || $lesson->module->course_id !== $course->id) { abort(404); }
```

## 2. HIGH — broken features & security gaps

### 2.1 Email verification links don't work
Route: `GET /verify-email/{token}` (routes/web.php:59). Controller: `VerifyEmailController::verify` reads `$request->query('token')` — the **query string**, not the route parameter. Links generated by `route('verification.verify', ['token' => $token])` put the token in the *path*, so `query('token')` is always null and every emailed link fails with "Invalid verification link." Since login requires `email_verified` (LoginController:36), non-Google users are likely locked out unless verified manually.

Also: `RegisterController` sends the verification email using the **`password_reset`** template (`sendTemplated($user->email, 'password_reset', ...)`) — wrong template, confusing content.

**Fix:** change the signature to `verify(Request $request, string $token)` (or read `$request->route('token')`), and use a proper verification template.

### 2.2 Newsletter subscribe endpoint 500s
Route uses `->middleware('throttle:newsletter')` (routes/web.php:389) but the limiter is registered as `newsletter-subscribe` (RouteServiceProvider:60). Laravel throws *"Rate limiter [newsletter] is not defined"* — every subscription attempt is a 500.

**Fix:** rename one to match. (This also suggests the endpoint has never been exercised — see "no tests" below.)

### 2.3 Promotions permanently block full payment & certificates
Chain of bugs:
- `CheckoutController::process` caps payment at `max:$effectiveBalance` (balance **after** promo discount).
- `Enrollment::effectivePrice()` / `isFullyPaid()` and `LencoPaymentService::updateEnrollmentPaymentStatus()` compare `amount_paid` against the **undiscounted** price and ignore `payments.discount_amount`.

Result: a student who redeems a promotion and pays the discounted balance in full can never reach `payment_status = 'completed'`; `certificate_blocked` stays `true` forever. Additionally, `used_count` increments on initiation (inside the checkout transaction) even if the Lenco payment later fails, and the increment condition (line 139) differs subtly from the discount-application condition (line 103).

**Fix:** account for completed payments' `discount_amount` when computing the outstanding balance (e.g. `totalPaid + totalDiscounts >= price`), and increment `used_count` only when the payment completes (in the webhook handler).

### 2.4 Password reset flow weaknesses
`ForgotPasswordController`:
- `'email' => 'required|email|exists:users,email'` returns a validation error revealing whether an account exists (enumeration). The resend-verification endpoint does this correctly with a neutral message — mirror that.
- Old reset tokens are not invalidated when a new one is requested; multiple live tokens accumulate in `remember_tokens`.
- `reset()` doesn't invalidate other sessions or check `users.status`, and the token is single-use only after success (fine) but there's no throttle on the POST `/reset-password` token-guessing path beyond the shared `password-reset` limiter (3/hour/IP — actually OK, keep it).

### 2.5 Webhook signature: probably validating with the wrong secret
`LencoPaymentService::validateWebhookSignature` HMACs with `LENCO_SECRET_KEY`, while `LENCO_WEBHOOK_SECRET` is defined in `config/services.php:27` but never read. If Lenco signs with the webhook secret, every signed webhook is rejected (401) and payments only complete via the 10-minute polling fallback — slow confirmations, and the signature check is effectively decorative. Verify against Lenco's docs and use the right key.

Also note: in non-production environments unsigned webhooks are accepted and processed (`LencoWebhookController:33` only enforces the signature in `production`). If staging ever points at real data, that's an open door. Prefer: always require a signature when a secret is configured.

### 2.6 CSRF exemption on logout
`VerifyCsrfToken::$except` includes `'logout'`. Any third-party page can force-logout your users (`<form action=.../logout method=post>`). Remove it — the logout form already includes `@csrf` in standard Blade layouts.

### 2.7 Session cookie not marked Secure
`config/session.php` → `'secure' => env('SESSION_SECURE_COOKIE', false)` and the committed production `.env` doesn't set it. On an HTTPS site the session cookie can still be sent over plain HTTP. Set `SESSION_SECURE_COOKIE=true` in production.

### 2.8 No MFA for privileged roles
Admin and Finance accounts (money verification, user management, role assignment) are protected by password only. Consider TOTP 2FA at least for role IDs 1, 2, 6.

## 3. MEDIUM — business logic & robustness

### 3.1 Registration fee flow creates orphan records
`RegistrationFeeController::initiateLencoPayment` creates the `Payment` and `RegistrationFee` rows **before** checking for an existing pending transaction; when the duplicate check trips, it returns leaving fresh orphaned `Pending` rows. Each retry adds more. Also the whole flow is not wrapped in a DB transaction (unlike checkout). Reorder: duplicate check → create records → call Lenco, inside one transaction.

### 3.2 Duplicate-payment guard relies on gap locks
`CheckoutController` uses `lockForUpdate()` on a row that may not exist; protection then depends on MySQL gap-locking behavior. A `UNIQUE` index on `lenco_transactions.reference` plus catching the duplicate-key exception is deterministic. Also `reference = 'EDU-{user}-{course}-' . time()` collides if two requests land in the same second — add randomness (`Str::random(6)`).

### 3.3 Quiz timing harshness & rounding
- `take()`: if time expired, the attempt is auto-graded **0** even if the student answered nothing yet (acceptable) — but `submit()` marks an over-time attempt `Abandoned` with score 0 *even when answers were provided seconds late*. Add a grace period (e.g. 30–60s) and grade what was submitted.
- `round($attempt->started_at->diffInMinutes(now()))` truncates to minutes; a 10-minute quiz submitted at 10:29 passes, at 10:31 fails — fine, but document it.

### 3.4 Hard-coded role IDs everywhere
`role_id` literals (1, 2 = admin, 3 = instructor, 4 = student, 6 = finance) appear in `User.php`, `RegisterController`, `GoogleController`, `CertificateController`, etc. A reseeded/altered `roles` table silently breaks authorization. Centralize in a `RoleEnum` or constants on the `Role` model.

### 3.5 Admin user management edge cases
- An admin can delete or demote **themselves** (`Admin/UserController::destroy/update`) — add a self-guard and a "last admin" guard.
- `update()` replaces all roles with a single role; a user who is both instructor and admin loses one silently (UI suggests single-role, but the schema is many-to-many).
- Deleting a user soft-deletes the `users` row but leaves enrollments/payments pointing at it — fine for accounting, but admin lists should surface "deleted user" rather than blank names.

### 3.6 Google OAuth account linking
`GoogleController` matches `orWhere('email', ...)` and logs in — acceptable for Google-verified emails, but combined with 1.3 it lets suspended users back in. Also `username` is set to the email (can collide with the `unique` rule on normal registration UX) and the random password is `bcrypt(uniqid())` — use `Str::random(32)`.

### 3.7 Payments listing relationship is convoluted
`User::payments()` builds a `whereIn` subquery against `students` inside a `hasMany` — works, but a `hasManyThrough(Payment::class, Student::class)` is correct and index-friendly.

### 3.8 Checkout `payment_method_id` map is hard-coded
`['mobile_money' => 2, 'lenco' => 6, 'bank_transfer' => 3]` (CheckoutController:94) depends on seeder row IDs. Look up by a `code` column on `payment_methods` instead.

## 4. UI / UX & responsiveness

Overall: the Tailwind + Alpine setup is solid — sticky nav with a real mobile menu, dashboard sidebar that collapses below `md` with localStorage persistence, toast component, empty-state component, `@tailwindcss/forms`. Specific gaps:

1. **Dark mode is dashboard-only.** `layouts/dashboard.blade.php` implements a `darkMode` toggle; the public layout (`app.blade.php`/`navigation.blade.php`) has zero `dark:` classes. A user who enables dark mode sees a white flash on every public page. Either extend it site-wide or scope the toggle's expectation clearly.
2. **Payment status UX is blind.** After initiating mobile money, the student is redirected to a generic "success" page (`payment.success`) that actually means *pending* — misleading name and message. There's a manual `/registration-fee/check` poller but nothing equivalent for course checkout, and no auto-refresh. Add a small JS poller (the session heartbeat pattern already exists in `routes/api.php`) hitting a payment-status endpoint, and rename the route/page to `payment.pending` when not yet confirmed.
3. **Accessibility:** only 23 of 154 views use any `aria-` attribute. The Alpine dropdowns (`@mouseenter`/`@mouseleave` menus in navigation.blade.php:43) are mouse-only — keyboard users can't open them. Add `aria-expanded`, focus/blur handlers, and visible focus rings.
4. **Responsiveness:** 90/154 views carry responsive prefixes; the main risk is wide admin/finance tables on phones. Spot-check `/admin/payments`, `/admin/enrollments`, `/finance/transactions` at 375px and wrap tables in `overflow-x-auto` (or use stacked card layouts on small screens).
5. **Quiz timer UX:** `remainingSeconds` is computed server-side; make sure the client counts down visibly and auto-submits before the server deadline (with the grace period from 3.3), otherwise students lose work at 0 score.
6. **Error pages:** `resources/views/errors` exists — good; confirm 403 messages ("Please complete at least a 30% deposit…") render in the styled layout, since several `abort(403, ...)` calls surface raw framework pages.

## 5. Workflow / engineering process

1. **Zero automated tests.** `tests/Feature` and `tests/Unit` are empty while the system moves real money. Highest-value first tests: checkout validation bounds, webhook → enrollment status transitions (incl. the promotion bug in 2.3), enrollment gating (`canAccessContent`), and the role middleware. PHPUnit is already configured — these are cheap wins.
2. **No CI.** Add a GitHub Actions workflow: `composer install`, `pint --test`, `phpunit`, `npm run build`. It would have caught the newsletter limiter 500 and the verification-link bug via one feature test each.
3. **Repo hygiene:** debug scripts at the root (`check-payments.php`, `check-payment-type.php`, `check-schema.php`), stray artifacts (`test-final.pdf`, `test-minimal.pdf`, `public_html_htaccess`). Move scripts to `scripts/` or delete; gitignore generated PDFs.
4. **Custom email pipeline:** PHPMailer + `email_queue` table + `queue:work --stop-when-empty` every 5 min works on shared hosting, but it duplicates Laravel Mail/queues and means up to 5-minute delays for verification emails (compounding 2.1). Consider Laravel's `database` queue driver + Mailable classes; same hosting requirements, far less custom code.
5. **Authorization style:** one policy (`CoursePolicy`) plus ad-hoc `roles()->whereIn(...)` checks scattered through controllers. Standardize on policies/gates so rules like "staff can view certificates" live in one place (`CertificateController::preview` and `::download` currently encode different staff sets: preview allows roles 1,2,3,6; download only 1,2 — likely unintentional drift).
6. **Scheduler dependency:** payment confirmation correctness depends on `lenco:poll-payments` running; verify the host's cron actually invokes `schedule:run` every minute and add monitoring/alerting (e.g. a `schedule:run` heartbeat log check) — if cron dies, payments silently stop confirming whenever webhooks fail.

## 6. What's already good

Worth keeping and building on:
- Consistent enrollment + 30%-deposit gating checks on every student content path (lessons, quizzes, assignments, downloads).
- Instructor controllers verify course ownership on every mutating action (`authorizeInstructor`).
- Lesson HTML is sanitized with HTMLPurifier on write *and* render; discussion content uses `e()` + `nl2br` — XSS posture is solid.
- Webhook handling is logged (`lenco_webhook_logs`) with signature status, plus a polling fallback command.
- Race-condition awareness: transactions + `lockForUpdate` on enrollment creation and quiz attempt numbering.
- Throttles on payment, enrollment, password reset, and verification resend endpoints.
